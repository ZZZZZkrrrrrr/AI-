## 三、前端已完成内容

前端已经从单个 HTML 页面升级为产品化控制台页面，包含以下模块：

1. 提示词工作台

4. 素材与输出  
   查看上传图片、提示词资产、本地生成视频文件和对应路径。

5. 系统设置  
   查看后端 API、千问视觉、豆包分析、libTV 桥接、数据库路径等运行状态。

## 四、核心业务链路

当前完整业务流程如下：

1. 用户在网页端上传提示词包  
   支持 `.docx`、`.txt`、`.md`、`.json`、`.csv`。其中 `.docx` 支持中文 Word 文档解析。

2. 用户上传商品图片  
   支持多张商品图片，前端展示缩略图。

3. 后端调用千问视觉 API  
   识别图片中的商品事实、外观、风格、人群、风险点，并辅助判断商品类别。

4. 后端调用豆包分析 API  
   根据提示词包内容动态拆解执行步骤。步骤数量不固定，可能是 5 步、10 步、15 步，按实际提示词包和模型结果决定。

5. 前端实时展示步骤状态  
   每一步都合并展示状态、输出结果和完成时间，避免先等全部步骤完成再统一输出。

6. 后端封装最终完整提示词  
   将提示词包、图片识别结果、商品补充信息、视频时长、画幅、类别等内容合并，生成可直接用于视频生成的最终提示词。

7. 用户确认最终提示词  
   最终提示词框支持自动填入，也支持用户手动粘贴、修改、复制。

8. 写入数据库并提交 libTV  
   点击后会写入 SQLite 数据库，同时注册 libTV 任务并调用本机 libTV 桥接服务。

9. libTV 返回结果  
   前端展示提交、执行、完成、失败等状态；成功后保存视频链接和本地输出信息。

## 六、数据库写入时机

当前数据写入逻辑如下：

1. 生成提示词阶段  
   写入 AI 步骤执行结果、最终提示词、任务事件、商品信息、类别识别结果。

2. 上传图片阶段  
   写入图片素材信息，包括文件名、格式、大小、路径或引用信息。

3. 点击“写库并提交 libTV”阶段  
   写入视频任务主记录、最终视频名称、libTV 节点名称、提交状态。

4. libTV 执行阶段  
   写入 libTV 外部任务 ID、执行状态、错误信息、视频链接、完成时间。

5. 异常阶段  
   写入错误记录和任务事件，前端同步展示失败原因。

## 七、视频命名与任务命名

当前链路已经将视频任务名称、数据库任务编号和 libTV 节点名打通。

命名目标是：最终视频名称需要包含序号、日期、类别，便于业务方追踪和归档。

当前逻辑中：

1. 类别优先来自图片识别结果。
2. 用户手动填写类别时，以用户填写为准。
3. 任务编号和 libTV 节点名使用标准化后的安全字符，避免中文、空格、特殊符号导致 libTV 或脚本异常。
4. libTV 节点名使用统一规则：`AIUGC-{任务编号}-video`。
5. 数据库中保存最终视频名称，前端任务看板和 libTV 任务页都从数据库读取，避免网页显示和数据库记录不一致。
