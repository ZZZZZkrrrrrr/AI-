# 抖音带货分类项目需求拆解

## 1. 需求目标

当前 AI 带货视频系统已经支持“提示词包 + 产品图片 -> AI 分析 -> 最终视频提示词 -> 数据库任务 -> libTV”。本次需求的核心是把《抖音带货分类》导图沉淀为系统可用的“带货赛道/品类/内容形态/案例参考”体系，让生成任务不只依赖手填类别，而是有明确的业务分类和内容策略依据。

## 2. 原始导图结构

导图共还原出 4 个一级分支、954 个节点：

- 全域
- 私域
- 挂车
- AIIP类

其中当前系统最应该优先落地的是“挂车”分支，因为它直接对应电商带货视频任务；“私域”和“AIIP类”可作为后续内容策略库和账号案例库。

## 3. MVP 需求范围

第一阶段建议只做系统可直接使用的能力：

1. 在提示词工作台增加“抖音带货分类”预设选择。
2. 扩展后端类别识别规则，让 AI 图片识别结果能落到更贴近导图的品类。
3. 任务编号继续使用 `日期-类别代码-序号`，但类别代码覆盖新增品类。
4. 生成最终提示词时，把所选分类作为明确业务上下文传给模型。
5. 保留手动输入能力，避免分类体系不完整时卡住任务。

## 4. 首批分类体系

优先接入这些品类：

- 男装
- 女装
- 童装
- 宠物用品
- 宠物拟人内容
- 食品
- 百货
- 美妆
- 保健品
- 家电
- 珠宝
- 五金建材
- DNA亲子鉴定
- 知识付费/私董会
- AI创业服务

## 5. 分类应携带的业务字段

每个分类建议沉淀为结构化配置：

- 分类名称
- 分类代码
- 匹配关键词
- 适合内容形式
- 目标用户画像
- 商品/服务选择方向
- 视频规格建议
- 常用人设
- 常用场景
- 对标账号或案例链接
- 风险/禁区

## 7. 验收标准

第一阶段完成后，应满足：

1. 用户在工作台可以直接选择男装、女装、宠物用品、食品、美妆等分类。
2. 不选择分类时，系统可根据商品名称、补充信息、图片识别结果自动推断分类。
3. 新生成任务的任务编号能体现新增类别代码。
4. 最终提示词生成时能明确使用当前分类上下文。
5. 原有手填类别、图片上传、提示词生成、写库、libTV 提交流程不被破坏。

## 8. 建议实施顺序

1. 先把分类预设接入前端工作台。
2. 扩展后端分类规则和任务代码规则。
3. 同步数据库优化脚本中的类别代码规则。
4. 构建并验证前端。
5. 用 3 个样例任务验证：女装、宠物用品、食品。
