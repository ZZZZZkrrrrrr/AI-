# AI 短视频带货工作台产品化与移动端设计方案

日期：2026-06-09  
项目：`D:\Organized\Projects\codex_project\workflow\01_apps\ai_prompt_video_studio`

## 1. 结论

这个项目现在已经具备“内部生产控制台”的雏形：能上传提示词包和商品图、调用视觉/文本模型、生成最终提示词、提交视频任务、查看任务与输出。但是它还不适合直接封装成普通用户使用的 App。

最务实的路线是：

1. 第一阶段先做“手机端网页 + PWA”，让用户通过浏览器访问、登录、创建任务、查看视频结果。
2. 第二阶段把后端从本机脚本和本地 SQLite 迁到云端服务，补齐账号、权限、额度、文件存储、任务队列、审计日志。
3. 第三阶段再用 Capacitor 封装 iOS/Android App。不要只做一个套壳 WebView，否则 Apple 审核和用户体验都有风险。
4. Android 如只面向 Google Play，也可以考虑 Trusted Web Activity；如果面向国内安卓应用商店，仍建议 Capacitor/原生壳并完成 APP 备案。

推荐优先级：手机端网页/PWA > Capacitor App > TWA Android 包 > React Native/Flutter 重写。

## 2. 当前项目现状

### 已有能力

- 前端：React + Vite 单页应用。
- 后端：Node.js 原生 HTTP 服务。
- 数据：SQLite、本地 runs 文件夹、本地输出目录。
- AI 能力：千问视觉、千问/豆包文本、Seedance/Seedream 相关接口预留。
- 工作流：选品资产、商品评分、账号资产、提示词工作台、批量生成、视频拼接、任务看板、libTV 任务、素材与输出、模型设置。
- 安全：已有登录、注册开关、会话 Cookie、用户归属字段。
- 构建：已完成第一轮拆包，主入口 JS 低于 500KiB，并有 `npm run app:health` 做结构、体积和 PWA 资产验收。
- 云端化：已新增公开探活接口 `/api/healthz` 和 `npm run cloud:check`，可以审计公开部署前的环境变量、本机路径、存储和 libTV 桥接风险。
- 合规页面：已新增隐私政策、用户协议、AI 生成内容说明和手机端入口，并有 `npm run compliance:check` 做页面、缓存和入口检查。
- 数据权利：已新增账号数据摘要导出和删除账号申请记录接口，并在手机端“我的 / 设置”提供“数据与账号”入口。
- AI 标识：单条视频创建页已支持下载 AI 生成内容证据包，记录任务、模型通道、输入摘要、输出摘要和人工复核清单。

### 主要问题

- 前端仍是桌面后台形态。`src/styles.css` 有 `body { min-width: 1120px; }`，虽然后面有移动端样式补丁，但核心页面仍保留大量宽表格和大网格。
- 入口过多。现在左侧导航按内部工作流分了很多模块，普通用户第一次打开会不知道先做什么。
- `src/App.jsx` 是一个超大文件，页面、业务规则、评分逻辑、合规逻辑、表单状态混在一起，后续迭代成本高。
- `server.js` 把静态服务、鉴权、API、任务、SQLite、文件服务、模型调用都放在一个文件里，不利于云端部署和多人使用。
- 运行依赖本机路径、SQLite、本机 libTV bridge、Python/ffmpeg。封装 App 后，用户手机不能直接访问这些本机资源。
- 当前服务端 CORS 是 `*`，Cookie、跨域、CSRF、生产域名安全策略还需要重做。
- 文件上传、AI 生成、视频任务都可能很重，需要队列、限流、失败恢复、额度计费。
- `npm run cloud:check` 当前无公开发布阻断项，但仍提示 libTV localhost、本机 Windows 路径、本地 runs 存储和干跑模式需要在云端化前处理。
- 面向真实用户仍需把内测版隐私政策、用户协议、AI 生成内容说明替换为正式主体、联系方式、第三方服务清单、完整账号删除执行、完整数据导出包、直播电商选品证据、应用备案和上架资料。

## 3. 封装方式怎么选

### 方案 A：手机端网页 + PWA

适合当前第一阶段。

优点：

- 改造成本最低，现有 React/Vite 可以继续用。
- 用户可以扫码访问，不需要安装。
- 适合快速验证：谁愿意上传商品图、谁愿意等待生成、谁愿意付费。
- 后续仍可作为 App 内 Web 页面复用。

需要做：

- HTTPS 域名。
- `manifest.webmanifest`、App 图标、启动页颜色、显示模式。
- Service Worker：缓存静态资源、提供离线错误页、处理版本更新提示。
- 移动端导航和页面重构。
- 上传、任务进度、视频预览、导出分享的移动端优化。

风险：

- iOS 上 PWA 安装入口不如 App Store 明显。
- 推送、后台任务、相册/文件权限等能力有浏览器差异。
- 对普通用户来说，“安装到桌面”的转化率通常低于应用商店下载。

### 方案 B：Capacitor 封装 iOS/Android

适合第二阶段以后。

优点：

- 可以保留大部分 Web 前端。
- 能接入相机、相册、文件系统、分享、推送、深链、生物识别等原生能力。
- 同一套业务 UI 可以同时发 Web、iOS、Android。

需要注意：

- Apple 不喜欢“只有网页套壳”的 App。必须有清晰的 App 级功能，例如本地素材导入、系统分享、推送、任务通知、离线草稿、相册保存、原生权限说明。
- 移动端 UI 需要真正按 App 设计，不能直接放桌面后台。
- 登录、支付、隐私、权限弹窗、文件访问、推送都要走商店规范。

### 方案 C：Trusted Web Activity

适合 Android 优先，且已有成熟 PWA。

优点：

- 对 Android PWA 上架 Google Play 更轻。
- Web 线上更新快，Android 壳相对简单。

限制：

- 主要解决 Android，不解决 iOS。
- 强依赖线上 PWA 质量、域名验证、`assetlinks.json`。
- 国内安卓应用商店生态下不一定是最顺的路径。

### 方案 D：React Native / Flutter 重写

短期不推荐。

只有在这些条件出现时才值得考虑：

- 需要大量本地视频编辑、离线处理、相册批量管理。
- WebView 性能已经无法满足。
- 团队有稳定的移动端工程能力。
- 产品模型已经验证，有足够预算重做。

## 4. 推荐移动端产品结构

当前桌面端是“全链路后台”。手机端应该改成“任务驱动的创作 App”。

### 4 个一级入口

底部导航建议只保留 4 个：

1. 首页：今日任务、生成进度、待处理、最近输出。
2. 创建：单条视频、批量任务、从商品库创建。
3. 素材：商品图、提示词包、账号资产、视频结果。
4. 我的：模型设置、额度、账号、安全、帮助。

不建议把“选品评分、商品库、账号资产、发布分发、数据回收、AI 复盘、任务看板、libTV 任务、素材输出、系统设置”全部放成手机端一级菜单。它们应该成为某个任务流程里的子页。

### 核心用户流程

#### 流程 1：创建单条视频

1. 选择商品：从商品库选择，或直接拍照/上传商品图。
2. 上传资料：商品图、提示词包、补充卖点、禁用词。
3. AI 识别：识别品类、卖点、风险点。
4. 合规预检：AI 标识、夸大宣传、资质/授权/价格/功效风险。
5. 生成脚本/提示词：显示可编辑结果。
6. 生成视频：进入任务进度页。
7. 审核导出：预览视频、复制提示词、下载/保存/分享。

#### 流程 2：批量生成

手机端不要做复杂 Excel 式大表格。建议改成：

- 批次卡片：显示批次名、商品数、通过预检数、失败数、预计耗时。
- 商品卡片：每个商品显示图片、标题、状态、风险、操作按钮。
- 批量操作：开始、暂停、重试失败、导出结果。
- 详细编辑放到单个商品抽屉页或独立页。

#### 流程 3：任务跟踪

所有异步任务统一成一套“任务中心”体验：

- 顶部显示当前运行任务。
- 卡片显示：状态、进度、当前步骤、预计剩余、错误原因。
- 操作：取消、重试、查看日志、打开结果。
- 长任务完成后支持站内通知和 App 推送。

#### 流程 4：素材和结果管理

手机端素材页建议按“可使用对象”组织，而不是按数据库表组织：

- 商品素材
- 提示词包
- 视频结果
- 账号人设
- 合规证据包

每个素材卡片都要有：预览、状态、来源、最近使用、操作。

## 5. 手机端页面设计细节

### 首页

目标：用户一打开就知道今天该做什么。

模块：

- 顶部：账号/空间、额度、通知。
- 今日概览：运行中、待审核、失败、已完成。
- 主要按钮：创建视频。
- 待处理队列：需要补资料、需要审核、生成失败。
- 最近输出：可直接预览/下载/分享。

### 创建页

目标：把复杂工作流收束成一个向导。

结构：

- 步骤条：商品资料 > AI 分析 > 提示词 > 视频 > 审核。
- 大上传区：拍照、相册、文件、历史素材。
- 输入项少而清楚：商品名、品类、卖点、限制词、目标平台。
- 每一步只有一个主按钮。
- 长文本编辑使用全屏编辑器，不塞在小卡片里。

### 任务详情页

目标：让等待过程可理解、可恢复。

结构：

- 状态标题：正在分析商品图 / 正在生成提示词 / 正在生成视频。
- 进度条和步骤列表。
- 当前模型、耗时、失败重试提示。
- 输出预览区。
- 底部固定操作：取消 / 后台运行 / 查看结果。

### 审核页

目标：让用户决定这个视频能不能发。

结构：

- 视频播放器。
- AI 生成内容标识状态。
- 卖点/价格/功效/授权风险提示。
- 提示词全文可复制、可编辑。
- 发布前清单：封面、标题、话术、商品信息、合规证据。

### 我的/设置

普通用户只需要：

- 账号与登录。
- 额度与套餐。
- 模型偏好。
- 通知设置。
- 隐私与协议。
- 数据导出/删除。

高级模型、API Key、数据库、libTV 桥接状态应隐藏在管理员模式。

## 6. 前端改造建议

### P0：先让手机能顺畅用

- 去掉全局 `body min-width: 1120px` 的桌面假设，改成页面局部容器控制。
- 建立 `AppShell`：桌面用侧边栏，手机用底部导航 + 顶部标题。
- 把宽表格改成卡片列表或响应式表格。
- 给核心页面设置移动端专用组件：`MobileHome`、`MobileCreateFlow`、`MobileTaskDetail`、`MobileAssetList`。
- 所有按钮点击区域至少 44px 到 48px 高。
- 输入框字体不小于 16px，避免 iOS 自动放大。
- 给底部固定按钮加 `safe-area-inset-bottom`。
- 视频预览固定 9:16 或 16:9 容器，避免布局跳动。

### P1：拆代码和性能优化

- 把 `App.jsx` 拆成：
  - `features/studio`
  - `features/batch`
  - `features/assets`
  - `features/settings`
  - `features/selection`
  - `shared/components`
  - `shared/api`
  - `shared/domain`
- 用 `React.lazy` / dynamic import 做路由级拆包。
- `xlsx` 这类大依赖只在批量导入页懒加载。
- 用虚拟列表处理大批量任务和商品卡片。
- 给长任务状态做本地缓存，刷新页面后能恢复。

### P2：PWA 能力

- 添加 `manifest.webmanifest`。
- 生成 192/512 图标、maskable icon、screenshot。
- 添加 Service Worker：
  - 静态资源预缓存。
  - API 不盲目缓存。
  - 任务详情可做 stale-while-revalidate。
  - 版本更新时给用户“刷新更新”提示。
- 添加离线页：网络断开时可查看历史任务，但不能创建新生成任务。
- 添加安装引导：Android 显示安装按钮，iOS 显示简短“添加到主屏幕”引导。

## 7. 后端产品化改造

### 目标架构

建议拆成这些部分：

- Web 前端：Vite 构建，部署到 CDN 或对象存储静态站。
- API 服务：Node/Express/Fastify/NestJS 均可，负责用户、素材、任务、配置。
- Worker 服务：执行 AI 调用、视频生成、拼接、文件处理。
- 数据库：PostgreSQL，替代生产 SQLite。
- 队列：Redis + BullMQ / RabbitMQ / 云厂商队列。
- 对象存储：阿里云 OSS / 火山 TOS / S3/R2，保存图片、Word、视频、证据包。
- 任务事件：SSE/WebSocket，移动端断线后可轮询恢复。
- 管理后台：模型配置、用户额度、任务审计、失败任务。

### 必须补齐的服务端能力

- 多租户：每条商品、素材、任务、输出都绑定 `tenant_id` 和 `user_id`。
- 额度：按生成次数、视频时长、模型 token、存储空间计费。
- 并发控制：每个用户/团队可同时运行的任务数。
- 文件上传：分片上传、类型校验、大小限制、病毒/内容扫描。
- 密钥管理：用户看不到平台 API Key；所有模型 Key 只存在服务端。
- 审计日志：谁上传、谁生成、谁导出、谁删除。
- 数据保留：视频、商品证据、直播合规记录要能按规则保留。
- 删除与导出：支持用户导出个人数据、删除账号数据。
- 安全：CORS 白名单、CSRF、防重放、限流、登录失败限制、安全 Cookie。

### libTV 依赖处理

当前 libTV 是本机桥接服务。产品化有三种选择：

1. 云端 Worker 安装 libTV/CLI，统一执行。
2. 自建渲染机集群，API 发任务，worker 拉取执行。
3. 如果 libTV 只能在用户本机运行，则做“桌面客户端/本地代理”，手机端只看任务和结果。

如果目标是普通用户手机 App，不能要求用户手机访问 `127.0.0.1` 或 Windows 本机脚本。

## 8. 合规与上架清单

### AI 生成内容

需要在系统里内置：

- 生成内容显式标识：视频画面、视频播放页、导出文件说明。
- 文件元数据/水印策略：能加就加。
- 用户协议说明：AI 生成内容标识方式、用户发布责任。
- 生成日志：内容编号、模型、时间、用户、素材来源。
- 用户不得删除/篡改标识的提示。

### 直播电商/短视频带货

需要产品内置：

- 商品资质核验字段：经营者、许可证、认证、合格证明、授权。
- 价格/功效/用户评价/荣誉资质等风险检测。
- 选品证据包：每个 SKU 的依据和记录。
- 直播/发布前清单。
- 至少三年保存策略，用于交易、直播回放、互动、商品详情、客服/售后等相关证据。

### APP/网页合规

中国大陆用户需要重点关注：

- 网站域名 ICP 备案。
- APP 备案。
- 隐私政策和用户协议。
- 个人信息收集最小化。
- 账号删除申请和数据导出入口。
- 第三方 SDK 清单。
- 应用商店隐私清单。
- AI 服务、算法、深度合成相关合规义务。

### App Store / Google Play

准备材料：

- 应用名、图标、截图、隐私政策 URL。
- App Store 隐私详情。
- Google Play Data Safety 表。
- 权限说明：相机、相册、文件、通知。
- 测试账号。
- 审核说明：核心功能不是网页套壳，包含任务通知、文件导入、相册保存、分享、离线草稿等 App 能力。

## 9. 迭代路线

### 第 0 阶段：产品边界确认，1 周

输出：

- 明确目标用户：商家、运营、达人、代运营团队。
- 明确首个付费场景：单条生成、批量生成、合规审核、素材库。
- 确定第一版不做的能力：复杂发布分发、全平台数据回收、深度 BI。

### 第 1 阶段：手机端 PWA MVP，2-4 周

目标：

- 用户能手机登录。
- 能创建单条视频任务。
- 能查看进度。
- 能审核和下载结果。
- 能查看历史任务。

开发任务：

- 移动端 AppShell。
- 底部导航。
- 创建向导。
- 任务卡片。
- 视频结果页。
- PWA manifest + service worker。
- 首页和我的页简化。

### 第 2 阶段：云端化，3-6 周

目标：

- 摆脱本机 SQLite 和本机 libTV bridge。
- 多用户可用。
- 文件和任务可持久化。

开发任务：

- PostgreSQL schema。
- 对象存储。
- Worker 队列。
- 任务状态机。
- 额度/限流。
- 管理后台。
- 线上日志与监控。

### 第 3 阶段：App 封装，2-4 周

目标：

- 用 Capacitor 封装 iOS/Android。
- 补齐原生能力。
- 完成内测分发。

开发任务：

- `@capacitor/core`、iOS、Android 项目。
- 相机/相册/文件/分享/推送插件。
- App 图标、启动图、权限文案。
- 深链和推送跳转任务详情。
- Store 截图和审核材料。

### 第 4 阶段：商业化，持续迭代

目标：

- 真实用户付费。
- 降低生成失败率。
- 提高视频可用率。

开发任务：

- 套餐和计费。
- 团队空间。
- 模板市场。
- 生成质量评分。
- 发布前合规分。
- 用户反馈闭环。

## 10. 第一批具体开发任务

建议优先做这些：

1. 新建移动端路由和底部导航，不动桌面端完整功能。
2. 把“创建单条视频”做成 5 步向导。
3. 把任务列表、批量任务从表格改成移动卡片。
4. 添加 PWA manifest、图标、Service Worker 和安装提示。
5. 把 `xlsx` 批量导入依赖懒加载，降低首屏 JS。

7. 服务端增加生产环境 CORS 白名单和持久化 session。
8. 设计 PostgreSQL schema 和对象存储目录。
9. 给视频输出加 AI 标识和证据包导出。
